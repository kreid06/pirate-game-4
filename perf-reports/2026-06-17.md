## Applied Changes

### Change 1 — Replace 8 blob-worker single-char `snprintf` with direct byte writes

**Files:** `server/src/net/websocket_server.c`  
**Functions:** `build_shared_blobs_from_snapshot`, `build_ships_blob_from_snapshot`  
**Evidence:** Each `snprintf(buf, cap, "[")` and `snprintf(buf+off, cap-off, "]")` call
invokes the full libc format-string path (parse `%`, clamp cap, write byte, null-terminate,
return count) for a single ASCII character. On Linux/glibc this costs ~50–100 ns per call
vs ~1 ns for a direct `buf[off++] = '['`. The blob worker fires once per tick, so at
30 Hz the two blob-builder functions together contained 7 such calls, totalling
~210 wasted calls/s in steady state and a larger spike on the first cold-path build.

Replaced calls (blob worker path):
| Line (before) | Old code | New code |
|---|---|---|
| 2061 | `int _to = snprintf(out->tmb_json, ..., "[");` | `out->tmb_json[0]='['; int _to=1;` |
| 2089 | `int _do = snprintf(out->ditem_json, ..., "[");` | `out->ditem_json[0]='['; int _do=1;` |
| 2107 | `int _co = snprintf(out->co_json, ..., "[");` | `out->co_json[0]='['; int _co=1;` |
| 2259 | `projectiles_offset += snprintf(..., "[");` | `out->projectiles_json[0]='['; projectiles_offset=1;` |
| 2279 | `projectiles_offset += snprintf(..., "]");` | direct bounds-checked write + explicit `'\0'` |
| 2348 | `ships_offset += snprintf(..., "[");` | `out->ships_json[0]='['; ships_offset=1;` |
| 2692 | `ships_offset += snprintf(..., "]");` | direct bounds-checked write + explicit `'\0'` |

Closing-bracket replacements match the pattern already used by the tmb/ditem/co builders
(lines 2085, 2103, 2121 use direct writes; the projectiles and ships builders were
inconsistent). Explicit `buf[offset] = '\0'` added after each closing write to preserve
the null-termination that the original `snprintf` provided implicitly.

### Change 2 — Replace per-client ship-slice opener `snprintf` with direct byte write

**File:** `server/src/net/websocket_server.c`  
**Function:** `websocket_server_update` — Phase 1 per-client AOI loop (line 12507 before)  
**Evidence:** `int _soff = snprintf(per_ship_json, PER_SHIP_BUF, "[")` ran once per
connected client per tick — up to `WS_MAX_CLIENTS (100) × 30 Hz = 3 000 calls/s`.
At 75 ns/call overhead that is ~225 µs/s of unnecessary format-string work on a
fully-loaded server.  

Replaced with:
```c
per_ship_json[0] = '[';
int _soff = 1;
```

## Proposed (needs human review)

### P1 — `schematics[128]` excluded from blob-worker snapshot

`SharedBlobSnapshot.players[100]` contains the full `WebSocketPlayer` struct, which
includes `PlayerBlueprint schematics[MAX_PLAYER_SCHEMATICS]` (128 × 16 bytes = 2 048 B
per player → ~200 KB for 100 slots). `build_shared_blobs_from_snapshot` never reads
`schematics` — only inventory slots, equipment, resource counts, and positional/health
fields are serialised. The 200 KB copy happens every tick at 30 Hz (6 MB/s of pure
cache churn). Fixing this requires either a `SlimPlayerSnapshot` struct that omits
schematics, or a selective field-by-field copy. The architectural change touches the
`SharedBlobSnapshot` definition, the `blob_worker_submit_snapshot` copy loop, and the
fallback `_snap` assembly — invasive enough to need a focused PR.

### P2 — `per_gs_pool` -Warray-bounds (carry-over from prior days)

`_goff` is a signed `int` that can exceed `PER_GS_BUF` if the guard macro
`_GS` miscounts; GCC's analyser sees the potential negative `size_t` and fires
four `-Warray-bounds=` warnings. The fix requires converting `_goff` to `size_t`
and auditing every `_GS`/`_MC1` expansion for correct unsigned saturation. Low
risk but non-trivial refactor.

### P3 — NPC blob: all 37 NPCs rebuilt every tick regardless of dirty state

`build_shared_blobs_from_snapshot` iterates all `MAX_WORLD_NPCS (512)` slots and
serialises every active NPC entry unconditionally. NPC positions only change when the
NPC moves (driven by its state machine). A `npc_dirty` flag per NPC (cleared after
serialisation, set on position/state write) would let the blob worker skip re-serialising
stationary NPCs. This requires adding the dirty flag to `WorldNpc` and setting it at
every mutation site — moderate cross-file change; deferred.

### P4 — `player_inventory_weight` called 3–5 times per player per tick

`player_inventory_weight` is called at lines 14037, 14228, 14344, 14498, 14674 in the
per-player tick processing loop. Each call iterates 16 inventory slots + 6 equipment
slots + 4 resource scalars (26 ops). With 100 players at 30 Hz that is 15 000 calls/s ×
~30 ns = ~0.45 ms/s. Hoisting one call per player per tick and passing the result to
the dependent paths would remove 4 redundant calls. Change requires local refactoring
within the per-player block; deferred for a focused PR since the five call sites are in
different nested scopes.
