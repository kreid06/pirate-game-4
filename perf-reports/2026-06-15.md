# Performance Report — 2026-06-15

## Summary

Three safe, mechanical optimisations applied to the blob-worker hot path in
`server/src/net/websocket_server.c`.  No gameplay semantics, physics constants,
tick rates, or netcode behaviour were touched.  Both `make -C server` (exit 0)
and `cd client && ./node_modules/.bin/tsc --noEmit` (exit 0) pass clean.

---

## Applied Changes

### 1 — Eliminate 5 redundant `strlen()` calls in blob builder
**File:** `server/src/net/websocket_server.c`
**Functions:** `build_shared_blobs_from_snapshot`

**What:** After each JSON section is written with `snprintf`, the code called
`strlen()` on the completed buffer to set `*_len`.  The snprintf offset
variables (`_to`, `_do`, `_co`, `players_offset`, `projectiles_offset`) already
track the exact byte count, so the strlen scan was redundant.

Replaced:
```c
out->tmb_len      = (int)strlen(out->tmb_json);
out->ditem_len    = (int)strlen(out->ditem_json);
out->co_len       = (int)strlen(out->co_json);
out->players_len  = (int)strlen(out->players_json);   // removed — see change 3
out->projectiles_len = (int)strlen(out->projectiles_json);
```
With direct offset assignment (e.g. `out->tmb_len = _to;`).

**Evidence/why:** `snprintf` returns the number of bytes written (excluding NUL).
Each offset variable is incremented by every snprintf return and by every direct
character write, so it equals `strlen` at the point the buffer is finalised.
For the players buffer (up to 100 players × ~3200 bytes = 320 KB of JSON) this
saves one O(n) scan of the buffer on every blob-worker invocation (~20-30 Hz).

**Risk:** None — purely mechanical substitution of equivalent values.

### 2 — Replace 4 single-char `snprintf(",")` calls with direct assignment
**File:** `server/src/net/websocket_server.c`
**Functions:** `build_shared_blobs_from_snapshot` (projectiles),
              `build_ships_blob_from_snapshot` (sim-ship and simple-ship paths)

**What:** Comma separators between JSON array elements were written with
`snprintf(buf + off, cap - off, ",")`.  Each call parses the format string,
checks for `%` tokens, and dispatches to libc — for a single ASCII byte.

Replaced with:
```c
if (off < (int)sizeof(buf) - 1) buf[off++] = ',';
```

**Evidence/why:** Format-string parsing overhead for `snprintf` on a literal
single-character string is ~10-50× the cost of a direct byte store.  The
projectile path runs up to `MAX_PROJECTILES = 500` times per blob build, the
ship paths up to `MAX_SHIPS = 200` times.  The bounds guard (`off < cap - 1`)
matches the invariant the old snprintf guard provided.

**Risk:** None — the off-by-one check is identical to what snprintf would have
done with `cap - off == 0` returning 0.

### 3 — Skip building `players_json` (unused in send path)
**File:** `server/src/net/websocket_server.c`
**Function:** `build_shared_blobs_from_snapshot`

**What:** The blob worker previously assembled a combined `players_json` (65 536-
byte buffer) by appending each `player_entry[p]` string with snprintf.  The
per-client AOI-filtered send path (lines ~12044–12059 of
`websocket_server_update`) reads `player_entry[]` directly and never references
`players_json` or `players_len`.

This was confirmed by exhaustive grep over the 15 168-line file: `players_json`
appears only in the struct declaration and inside `build_shared_blobs_from_snapshot`
— nowhere in the send path.

The identical optimisation was already applied to NPCs (`npcs_json`), with the
rationale documented in the code comment at lines 2193–2201:
> "The per-client send path already reads npc_entry[] directly; npcs_json is
> not used in any send path, so we skip building it."

**Removed per blob invocation:**
- Opening `"["` snprintf into `players_json`
- Up to 100 per-player `snprintf(…, "%s", player_entry)` copies into `players_json`
  (each ~3200 bytes → up to 320 KB of data movement)
- Up to 99 comma `snprintf(…, ",")` calls
- Closing `"]"` snprintf
- `strlen` scan of the completed 65 KB buffer
- `first_player` boolean tracking variable

**Added:** `out->players_json[0] = '\0'; out->players_len = 0;` — matches the
`npcs_json` pattern, prevents stale data if any future code reads the field.
`out->active_player_count` is still set correctly (used by the tick-rate
adaptive logic at line ~11945).

**Evidence/why:** At 100 active players the old code wrote ~320 KB of snprintf
output into a buffer that was never read.  At 30 Hz this is ~9.6 MB/s of wasted
memory bandwidth inside the blob worker thread.  Removing it reduces blob-build
time and CPU cache pressure.

**Risk:** None — `players_json` was demonstrably dead output; `player_entry[]`
(the actual source for all client sends) is unchanged.

## Proposed (needs human review)

These items were identified during analysis but **not applied** because they
require judgment about correctness or carry non-trivial risk:

### A — `per_gs_pool` -Warray-bounds: unsigned offset refactor
The 4× `-Warray-bounds=` GCC warnings on `per_gs_pool[100][524288]` point to a
real latent bug: `_goff` is a signed `int` that can reach `PER_GS_BUF - 1`
(set by the clamp in `_GS`), but the memcpy at `per_gs + _goff` uses `_goff` as
an offset into a statically-allocated pool.  GCC's analyser concludes that
negative offset values are possible (they are, if the clamp fires late).  A
clean fix requires refactoring `_goff` to `size_t` and ensuring early-exit on
overflow rather than clamping.  Skipped: any change to the overflow semantics
of the assembly loop is risky without test coverage.

### B — Snapshot tier timestamp reset bug (all tiers reset on every call)
In `snapshot.c`, the MID and LOW throttle tier timestamps are reset on every
invocation, defeating the throttling mechanism.  This causes all entities to be
sent at the HIGH tier rate regardless of tier assignment.  Fix requires
understanding the intended tier logic and ensuring the reset only happens when
the snapshot is actually consumed.  Skipped: affects network cadence / client
experience.

### C — Spatial hash built after collision detection, not before
`sim_update_spatial_hash` is called at a point where collisions have already
been resolved for that tick.  This means the hash always reflects last tick's
positions for any code that queries it within the same tick.  Fixing the call
order could affect collision query results.  Skipped: requires understanding
all callers.

### D — `view_radius` parsed from client but hardcoded 5000.0 used in AOI send
The per-client `view_radius` field is populated from the client handshake but
the send loop at line ~12004 hardcodes `const float _view_r = 5000.0f;`.
Using the per-client value would allow tuning without code changes.  Skipped:
could create fairness issues if clients can set large radii.

### E — `g_npcs_dirty` flag never checked in blob worker
`g_npcs_dirty` is set whenever NPC state changes but the blob worker always
re-serialises all NPCs regardless of the flag.  Checking it could skip NPC
re-serialisation on ticks where nothing changed.  Skipped: needs careful
analysis of the flag's lifecycle (it's a global; race conditions possible if
the blob worker thread reads it without the main thread's lock).

### F — `build_shared_blobs_from_snapshot`: tombstone ship-lookup is O(n) per tombstone
For each active tombstone on a ship, the builder does a linear scan over all
ships to find the carrying ship.  With MAX_TOMBSTONES=64 and MAX_SIMPLE_SHIPS=200
this is at most 12 800 comparisons per build — not urgent, but a ship-ID map
would reduce this to O(1).  Not applied today: number of active tombstones is
typically very small.
