## Applied Changes

### Change 1: SnapShipLut `aoi_idx[]` — O(1) on-ship world position for player/NPC AOI

**File:** `server/src/net/websocket_server.c`  
**Functions:** `SnapShipLut`, `build_ships_blob_from_snapshot`, `build_shared_blobs_from_snapshot`

**What:** While building ship JSON, record each ship's AOI table index in `lut->aoi_idx[ship_id]`. Player and NPC world-position computation for AOI filtering uses `snap_ship_lut_aoi_pos()` instead of scanning `aoi_ship_count` entries per entity.

**Evidence:** With 200 ships and 512 NPCs, worst-case inner scans were O(entities × ships). At 37 NPCs and 2 ships the per-tick saving is small but the lookup is now constant-time and removes redundant work that scales with fleet size.

**Risk:** None. `snap_ship_lut_aoi_pos` validates `aoi_ship_id[idx] == sid` before returning coordinates.

*(Backlog from 2026-06-20; not yet merged to develop.)*

### Change 2: Direct `players[p]` for grapple carry weight in blob worker

**File:** `server/src/net/websocket_server.c` — `build_shared_blobs_from_snapshot`

**What:** Replaced an O(WS_MAX_CLIENTS) scan to find the live player slot matching `snap->players[p].player_id` with a direct index: blob snapshot slot `p` maps to `players[p]` and `grapple_hooks[p]`.

**Evidence:** Up to 100 comparisons per active player per blob build eliminated. Blob worker and main thread use the same slot indexing for player/grapple state.

**Risk:** None when `players[p].player_id == snap->players[p].player_id` (guaranteed by `copy_player_to_blob` from the same slot).

*(Backlog from 2026-06-20.)*

### Change 3: Player entry length from `snprintf` return; `memcpy` grapple append

**File:** `server/src/net/websocket_server.c` — `build_shared_blobs_from_snapshot`

**What:** Track `_pelen` from the initial `snprintf` return value. Append grapple JSON via `memcpy` instead of `strlen` + `strncat`.

**Evidence:** Removes 2–3 redundant scans of the ~3 KB player entry buffer per grappling player per tick.

**Risk:** None. Same JSON output; bounds checks preserved.

*(Backlog from 2026-06-20.)*

### Change 4: Bounded `dynamic_companies` memcpy in blob submit + sync fallback

**File:** `server/src/net/websocket_server.c` — `blob_worker_submit_snapshot`, sync fallback in `websocket_server_send_game_state`

**What:** Copy only `dynamic_company_count` active company slots (or zero the buffer when count is 0) instead of `memcpy(..., sizeof(dynamic_companies))` (64 entries).

**Evidence:** `DynamicCompany` is ~80 B; typical `dynamic_company_count` is 0–few. Saves ~5 KB per tick on the submit hot path.

**Risk:** None. JSON builder already iterates `_dcc = dynamic_company_count` and skips inactive entries.

*(Backlog from 2026-06-20.)*

### Change 5: Skip `copy_player_to_blob` for inactive player slots

**File:** `server/src/net/websocket_server.c` — `blob_worker_submit_snapshot`, sync fallback

**What:** Only call `copy_player_to_blob` when `players[slot].active`; otherwise set `job.players[slot].active = false`.

**Evidence:** `BlobPlayer` is ~700 B (no schematics). With 0 players online, this avoids ~70 KB of field copies per tick on the submit path. Blob builder already skips inactive entries.

**Risk:** None. Inactive slots are explicitly marked inactive so stale data cannot leak into JSON.

---

### Change 6: Reuse cached `player_world_x/y` for viewer AOI origin in send loop

**File:** `server/src/net/websocket_server.c` — `websocket_server_send_game_state` phase 1

**What:** Per connected client, use precomputed `shared_blob_cache.player_world_x/y[vslot]` (includes on-ship position via Change 1) instead of re-scanning `aoi_ship_id[]` when `parent_ship_id != 0`.

**Evidence:** Removes O(aoi_ship_count) work per connected client per tick. With 100 clients and 200 ships, that is up to 20 000 comparisons/s at 30 Hz.

**Risk:** None. Cache is built in the same blob frame and already used for player AOI filtering.

## Proposed (Needs Human Review)

### A. `per_gs_pool` -Warray-bounds: signed offset wrap

`per_gs_pool[100][524288]` is written via `_goff` (int). When `_goff` exceeds `PER_GS_BUF`, size arithmetic can wrap. The `_GS` macro guards `snprintf` but `_MC1` memcpy sites still trigger compiler warnings. Fix requires `size_t _goff` with a hard cap before every write. Deferred: correctness-critical send loop.

### B. Player entry dirty-flag cache

NPC entries use `npc_key[]` to skip `snprintf` when unchanged. Player entries (~3 KB, inventory + grapple) rebuild every tick. A `player_key[]` cache would help docked/AFK players but must include grapple state and derived `speed_mult`/`can_sprint`. Deferred: worker-thread/global coupling needs audit.

### C. Tombstone / dropped-item full-array memcpy on blob submit

`memcpy(tombstones, sizeof(tombstones))` and `memcpy(dropped_items, sizeof(dropped_items))` copy fixed 64 + 256 slots regardless of active count. Arrays are sparse; bounded copy needs slot-compaction audit or active-count tracking. Deferred.

### D. `view_radius` hardcoded to 5000

Client-reported view radius is parsed but send path uses `const float _view_r = 5000.0f`. Changing this affects AOI fairness — gameplay decision.

### E. `placed_structure_count` linear scans in movement hot path

Multiple O(structures) scans per moving player for dock/shipyard lookup. Structure-ID hash would help but touches data layout.

### F. `handle_island_collisions` / `handle_ship_collisions`

Correctness-critical physics; no safe automatic change.

### G. UDP `snapshot.c` delta baseline O(n²) entity lookup

UDP snapshot path not active at runtime; deferred.
