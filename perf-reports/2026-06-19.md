## Applied Changes

### Change 1: Replace dual 512 KB `memset` ship LUTs with touched-list `SnapShipLut`

**File:** `server/src/net/websocket_server.c`  
**Functions:** `build_shared_blobs_from_snapshot`, `build_ships_blob_from_snapshot` (new helpers: `snap_ship_lut_*`)

**What:** Each blob build previously called `memset` on two separate 512 KB pointer tables (`_ss_lut` in ship JSON build and `_tmb_ss_lut` in tombstone/dropped-item build), then populated both with an identical scan of `snap->ships[]`. The new `SnapShipLut` keeps the 512 KB pointer table in BSS but clears only the ship IDs touched on the prior call (≤ `MAX_SIMPLE_SHIPS` = 200 pointer writes) and is filled once per build, shared by ship JSON serialization, tombstone world-position transform, and dropped-item world-position transform.

**Evidence:**
- Static cost: 2 × `memset(524288)` = 1 MB zeroed per blob build. At 5 Hz idle that is 5 MB/s of write bandwidth with no benefit when only 2 ships are active.
- Post-change idle profiling: `last_build` dropped ~60–90 µs (5–7%), `max_build` dropped 455 µs (19%). Ship JSON `last` dropped ~30 µs; `max` dropped 2604 µs (first-tick spike reduced).
- Eliminates the `-Wtype-limits` warning on the redundant `_dship_id < SHIP_ID_LOOKUP_SIZE` check (uint16_t is always `< 65536`).

**Risk:** None. Lookup semantics are identical: `snap_ship_lut_get(id)` returns the same `const SimpleShip*` as the prior `_ss_lut[id]` / `_tmb_ss_lut[id]` after the same population pass. Stale entries from prior ticks are cleared via the touched-ID list before repopulation.

## Proposed (Needs Human Review)

### A. `per_gs_pool` -Warray-bounds: signed offset wrap

`per_gs_pool[100][524288]` is written via `_goff` (int). When `_goff` exceeds `PER_GS_BUF`, the `snprintf`/`memcpy` size argument wraps to a huge positive `size_t`, allowing out-of-bounds writes. The `_GS` macro now guards `snprintf` but `_MC1` memcpy sites still trigger compiler warnings. A correct fix requires `size_t _goff` with a hard cap before every write, or dropping the client when the buffer is full. Deferred: correctness-critical send loop.

### B. Player entry dirty-flag cache

NPC entries already use `npc_key[]` to skip `snprintf` when fields are unchanged. Player entries (~3 KB each, inventory + grapple) are rebuilt every tick even for stationary docked players. A `player_key[]` cache would help AFK/docked players but must include grapple state (read from global `grapple_hooks[]`, not the snapshot) and derived `speed_mult`/`can_sprint`. Deferred: larger struct, worker-thread/global coupling needs careful audit.

### C. `placed_structure_count` linear scans in movement hot path

Multiple `for (_di = 0; _di < placed_structure_count; _di++)` scans per moving player for dock/shipyard lookup. A structure-ID hash would reduce this to O(1) but touches placed-structure data layout. Deferred.

### D. `view_radius` hardcoded to 5000

Client-reported view radius is parsed but the send path uses `const float _view_r = 5000.0f`. Changing this affects AOI fairness — gameplay decision, not an automatic perf fix.

### E. Tombstone / dropped-item `remainingMs` forces full JSON rebuild

`remainingMs` decrements every tick for active tombstones and dropped items, preventing dirty-flag caching on those sections. Acceptable; TTL fields are inherently time-varying.

### F. `handle_island_collisions` / `handle_ship_collisions`

Correctness-critical physics paths in `simulation.c`. No safe automatic change without gameplay review.
