## Applied Changes

### Change 1: Early-out player-only collision handlers when no sim players

**File:** `server/src/sim/simulation.c`  
**Functions:** `handle_player_ship_collisions`, `handle_player_boulder_collisions`, player-vs-ship CCD block in `sim_handle_collisions`, player loop in `handle_projectile_collisions`

**What:**
- Return immediately from `handle_player_ship_collisions` and `handle_player_boulder_collisions` when `sim->player_count == 0`.
- Wrap the swimming-player vs ship-hull CCD pre-pass in `if (sim->player_count > 0)`.
- Wrap the projectile-vs-player inner loop in `if (sim->player_count > 0)`.
- Add `if (sim->projectile_count == 0) return;` at the top of `handle_projectile_collisions`.

**Evidence:** Profiling runs with 0 websocket clients also have `sim->player_count == 0`, yet the prior code still entered player collision handlers every tick. `handle_player_boulder_collisions` alone nests `player_count × ISLAND_COUNT × resources` island/boulder checks. With ~150 ghost ships and 0 players this was pure overhead. Early-outs preserve semantics: when `player_count == 0` the loops had zero iterations anyway.

**Risk:** None. Guards only skip empty loops; behavior unchanged whenever players exist.

---

### Change 2: Skip empty spatial-hash insert loops

**File:** `server/src/sim/simulation.c`  
**Function:** `sim_update_spatial_hash`

**What:** Only iterate players/projectiles when `sim->player_count > 0` / `sim->projectile_count > 0` before calling `spatial_hash_add_player` / `spatial_hash_add_projectile`.

**Evidence:** The spatial hash is fully cleared with `memset` each tick (~1.6 MB for 64×64 cells). Player and projectile insert loops were still entered and checked `i < 0` bounds even when counts were zero. Skipping them removes redundant loop overhead on idle-player profiles.

**Risk:** None. Same entities inserted when counts are non-zero.

---

### Change 3: `(size_t)` remaining-space guards on tombstone broadcast snprintf

**File:** `server/src/net/websocket_server.c`  
**Functions:** tombstone spawn broadcast, `send_tombstone_items_to_client`

**What:** Inventory slot append snprintf calls now use `(size_t)(sizeof(msg) - (size_t)off)` instead of signed `sizeof(msg) - off`.

**Evidence:** Same signed/unsigned wrap class documented in prior reports (NPC combined-buffer bug, ship module snprintf guards). Tombstone messages are built on player death (cold path) but use the same anti-pattern; closing it prevents buffer-size wrap if `off` ever exceeds `sizeof(msg)`.

**Risk:** None. Defensive bounds consistency only.

## Proposed (Needs Human Review)

### A. `per_gs_pool` -Warray-bounds: oversized payload silent truncation

When `_goff + len` exceeds `PER_GS_BUF`, payloads are silently truncated. Fix needs full `size_t _goff` hard-cap audit across the send assembly loop.

### B. Prebuild shared proj/tmb/ditem/co JSON tail once per tick

Per-client GAME_STATE assembly re-memcopies identical projectile/tombstone/ditem/company sections for every connected client. High impact at 20+ clients; requires careful JSON field-order preservation.

### C. Ship JSON dirty-flag cache (player/NPC pattern)

Per-ship `snprintf` rebuild runs every tick even when sim state unchanged. Needs stale-field audit (modules, claim flags, mass) before skipping.

### D. `sim_update_spatial_hash` full 1.6 MB memset every tick

Incremental dirty-cell clearing needs collision correctness audit.

### E. Grapple-target player dirty key extension

`player_is_grapple_target()` forces full player reserialize every tick; anchor coords could be folded into `player_key`.

### F. Skip blob-worker submit when zero websocket clients connected

Would save ~1 ms/tick on idle ghost-fleet servers but needs a reliable connected-client counter maintained on every connect/disconnect path.

### G. UDP `snapshot.c` delta baseline O(n²) entity lookup

UDP snapshot path not active at runtime; baseline id→index LUT would help if re-enabled.
