## Applied Changes

### 1. `server/src/sim/simulation.c` — Selection sort → insertion sort in tick loop

**What:** `sim_update_ships`, `sim_update_players`, and `sim_update_projectiles`
each sort their entity array by ID every tick to guarantee deterministic
simulation order.  The previous implementation used a selection sort (the
O(n²) compare-every-pair variant).  Replaced with insertion sort.

**Evidence / math:**

Entity IDs come from a single monotonically-increasing global counter
(`allocate_entity_id`; static `next_id` starting at 1).  New entities are
appended at the tail of their array (`sim->ships[sim->ship_count]`, etc.).
When an entity is removed the array is compacted with `memmove`, which
preserves relative order.  Therefore the array is already sorted in the
common case (only the most-recently added entity may be out of place if it
was allocated with an ID that interleaved with another entity type, which
the global counter prevents).

| Array | MAX size | Selection sort (always) | Insertion sort (already sorted) |
|---|---|---|---|
| ships | 200 | ~20,000 comparisons | ~200 |
| players | 100 | ~5,000 | ~100 |
| projectiles | 500 | ~125,000 | ~500 |

At a 20 Hz tick rate:

| Entity | Before (cmp/s) | After (cmp/s) | Saved |
|---|---|---|---|
| Ships (worst-case) | 400,000 | 4,000 | ~99% |
| Players (worst-case) | 100,000 | 2,000 | ~98% |
| Projectiles (worst-case) | 2,500,000 | 10,000 | ~99.6% |

Practical savings are smaller (typical live counts are well below MAX), but
insertion sort is strictly never worse than selection sort for any input.

**Safety:** The sorted result is identical for any input.  Only the algorithm
changed.  Insertion sort is stable; selection sort is not — but since entity
IDs are unique the tie-breaking behaviour is moot.

**Files changed:** `server/src/sim/simulation.c` (lines 322–332, 497–506, 529–538)

### 2. `server/src/net/snapshot.c` — Replace `(void)buffer_size;` with bounds check

**What:** `snapshot_generate_for_player` accepted a `buffer_size` parameter
that was immediately discarded with `(void)buffer_size;`.  This left no guard
against callers passing an undersized buffer.  Replaced with an explicit
minimum-size check:

```c
if (buffer_size < sizeof(struct SnapshotPacket)) {
    log_error("snapshot_generate_for_player: buffer too small (%zu < %zu)",
              buffer_size, sizeof(struct SnapshotPacket));
    return -1;
}
```

**Evidence:** Static analysis: the original code contained a parameter that
was explicitly suppressed, meaning the function could write up to
`sizeof(struct SnapshotPacket)` bytes into a buffer it never verified was
that large.  All current callers pass a correctly-sized buffer so behaviour
is unchanged at runtime.

**Safety:** Pure defensive hardening; no logic or data changed.

**Files changed:** `server/src/net/snapshot.c` (line 101)

## Proposed — Needs Human Review

The following issues were identified but **not applied** because they involve
netcode behaviour, larger refactors, or signed/unsigned arithmetic whose
semantics need verification.

### A. `websocket_server.c` — `per_gs_pool` negative-offset buffer overrun (4× `-Warray-bounds=`)

**Location:** `websocket_server_update`, around the GAME_STATE assembly loop
(~line 11389).

**Issue:** `_goff` is an `int` accumulated via repeated
`_goff += snprintf(per_gs + _goff, PER_GS_BUF - _goff, ...)`.  When the
buffer approaches full, `snprintf` returns its **would-have-written** length
(not the truncated length).  This can push `_goff` above `PER_GS_BUF`.  The
next call then computes `PER_GS_BUF - _goff < 0`, which when implicitly cast
to `size_t` becomes a huge positive number — causing snprintf to write past
the 512 KB `per_gs_pool` buffer.  GCC's VRP analysis detects this and reports
four `-Warray-bounds=` warnings.

**Proposed fix:** After each bare `snprintf` accumulation, clamp `_goff` to
`PER_GS_BUF - 1`.  However, a naïve post-snprintf clamp (`if (_goff >=
PER_GS_BUF) _goff = PER_GS_BUF - 1;`) interacts with the `_MC1` macro's
signed-integer overflow path in a way that actually doubles the warning count
rather than eliminating it — GCC's range analysis expands the possible value
set for `_goff` through the new branches.

The correct fix requires refactoring this section to use either:
- An `unsigned` offset variable (prevents the negative-index code path), or
- A dedicated safe-append function that returns a clamped offset, or
- Changing `_MC1` to explicitly assert `len >= 0` so GCC can narrow the
  range.

This is more invasive than a single-line change and needs human review.

### B. `snapshot.c` — Tier timestamp reset defeats snapshot throttling

**Location:** `snapshot_generate_for_player`, lines 278–281.

```c
for (int tier = 0; tier < AOI_TIER_COUNT; tier++) {
    player_state->last_snapshot_time[tier] = current_time;
}
```

This resets `last_snapshot_time` for **all** tiers on every snapshot call,
regardless of which tier was actually sent.  The `should_send_snapshot_for_tier`
function checks `(current_time - last_time) >= interval_ms`; resetting
timestamps for tiers that weren't sent means those tiers will never be
throttled — they always appear "ready" on the next call.  MID and LOW tier
entities end up sent at HIGH-tier frequency (30 Hz), wasting ~6×–3× the
intended network bandwidth for distant entities.

**Not applied:** This is a netcode behaviour change (snapshot send frequency)
affecting what players perceive.  The current behaviour may be intentional
(always send all subscribed entities at maximum rate), or it may be a bug.
Human confirmation required before throttling is activated.

### C. `simulation.c` — Ship module healing loops: 3 passes → 1 pass

**Location:** `sim_update_ships`, per-ship module section.

Three separate loops iterate `ship->module_count` each tick:
1. Passive plank healing (lines ~380–392)
2. Passive non-plank module healing (lines ~396–420)
3. Plank count for hull integrity (lines ~436–444)

These could be merged into a single pass (reducing cache-line pressure for
large module counts).  Not applied because:
- The merge has subtle ordering: the scaffolded early-out (line ~425) sits
  between loop 3 and loop 4, so merging all four requires careful placement
  of the scaffolded check.
- The `MODULE_STATE_REPAIRING | MODULE_STATE_DAMAGED` clearing condition for
  non-plank modules differs from planks; the merged logic must preserve this.

### D. `simulation.c` — Spatial hash rebuilt after (not before) collisions

The `sim_update_spatial_hash` call in `sim_step` runs **after**
`sim_handle_collisions`.  The ship-ship and projectile-ship collision
handlers therefore cannot use the spatial hash for broad-phase culling —
they fall back to O(n²) and O(P×S) full scans.  Moving
`sim_update_spatial_hash` before `sim_handle_collisions` would enable
cell-based broad phase, potentially cutting collision work by 90%+ at scale.
Not applied because: the hash is populated with positions from the current
tick **before** physics integration; moving the hash rebuild before
collisions would use the pre-integration positions.  The integration step
inside `sim_update_ships` (which runs before collisions) changes positions,
so the hash must be rebuilt after integration.  A redesign to build the hash
**after** integration but **before** collisions needs human assessment of
tick-order correctness.

### E. `websocket_server.c` — `view_radius` parsed but unused in AOI send

The server parses `player.view_radius` from client input (stored in
`WebSocketPlayer.view_radius`) but the send loop uses a hardcoded
`_view_r = 5000.0f`.  Using the client-reported radius would dynamically
reduce the AOI broadcast for players on zoomed-in views, lowering network
traffic.  Not applied because this is a fairness/anti-cheat boundary: the
client should not be trusted to self-report a radius that reduces what it
receives (it could claim radius=0 to avoid seeing enemies).  A server-side
cap formula is needed.

### F. `websocket_server.c` — `g_npcs_dirty` flag set but never consulted in blob worker

`g_npcs_dirty` is set when NPC state changes but the blob worker
unconditionally rebuilds all NPC `npc_entry[]` JSON every tick.  Checking
this flag and skipping NPC JSON rebuild on clean ticks would save up to
512 × `snprintf` calls per tick when NPCs are idle.  Not applied because it
requires caching `npc_entry[]` across ticks (correct snapshot semantics for
blob output handoff between worker and main thread) and needs careful
mutex/double-buffer handling.

### G. Pre-existing `fread` return-value warnings (4 instances)

`world_save.c`, `player_persistence.c`, and `admin_api.c` each call `fread`
without checking the return value.  A short read (e.g. partial disk write,
file truncation) would silently use uninitialised or stale buffer contents.
Not applied because the correct fix (handling partial reads or returning
errors) requires understanding the error recovery strategy for each call
site.
