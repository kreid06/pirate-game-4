## Applied Changes

### Change 1: `BlobTombstone` slim snapshot â€” drop unused inventory from blob submit

**File:** `server/src/net/websocket_server.c`  
**Functions:** `SharedBlobSnapshot`, `copy_tombstones_to_blob`, `blob_worker_submit_snapshot`, sync fallback

**What:** Replaced `Tombstone` (includes full `PlayerInventory` ~300 B) in `SharedBlobSnapshot` with `BlobTombstone` containing only the fields the broadcast JSON uses (id, position, ship attachment, owner name, spawn time, active). **Evidence:** `build_shared_blobs_from_snapshot` never reads `tombstones[_ti].inventory`. Full-array memcpy moved ~25 KB/tick (64 Ă— ~390 B) including inventory data the worker never touches. `max_build` dropped from 2238â€“4381 Âµs to 1528â€“1715 Âµs on idle profiling.

**Risk:** None. Death-loot inventory is handled on interaction paths, not in GAME_STATE tombstone JSON.

### Change 2: `player_key[]` dirty-flag cache for player entry JSON

**File:** `server/src/net/websocket_server.c` â€” `SharedBlobOutput`, `build_shared_blobs_from_snapshot`

**What:** Added `player_key[WS_MAX_CLIENTS]` mirroring the existing `npc_key[]` pattern. Compares every serialized field (including inventory, grapple state, and derived-speed inputs) before rebuilding the ~3 KB `player_entry[]` string.

**Evidence:** Same approach as NPC cache (already in production). Saves ~3 KB snprintf + inventory formatting per stationary player per tick. No measurable idle-server delta (0 players online) but scales linearly with connected AFK/docked players.

**Risk:** None when key fields match JSON output (includes grapple hook state read from `grapple_hooks[p]`, same as prior unconditional rebuild).

### Change 3: Skip redundant ground-player AOI world-position update

**File:** `server/src/net/websocket_server.c` â€” `build_shared_blobs_from_snapshot`

**What:** When `parent_ship_id == 0` and the player entry is not dirty, reuse cached `player_world_x/y` instead of reassigning from unchanged `x/y`. On-ship players still recompute world position every tick (ship may move without player field changes).

**Evidence:** Mirrors the NPC on-ship rationale documented in prior reports. Saves two float writes per stationary ground player per tick.

**Risk:** None. Ground players only skip reassignment when every serialized field (including `x/y`) is unchanged.

## Proposed (Needs Human Review)

### A. `per_gs_pool` -Warray-bounds: signed offset wrap

`per_gs_pool[100][524288]` is written via `_goff` (int). When `_goff` exceeds `PER_GS_BUF`, size arithmetic can wrap. Fix requires `size_t _goff` with a hard cap before every write. Deferred: correctness-critical send loop.

### B. `dropped_items` full-array memcpy on blob submit

`memcpy(dropped_items, sizeof(dropped_items))` copies 256 sparse slots (~10 KB) every tick. Unlike tombstones, all dropped-item fields are used in JSON â€” bounded copy needs active-slot compaction audit.

### C. `view_radius` hardcoded to 5000

Client-reported view radius is parsed but send path uses `const float _view_r = 5000.0f`. Changing this affects AOI fairness â€” gameplay decision.

### D. `placed_structure_count` linear scans in movement hot path

Multiple O(structures) scans per moving player for dock/shipyard lookup. Structure-ID hash would help but touches data layout.

### E. `handle_island_collisions` / `handle_ship_collisions`
