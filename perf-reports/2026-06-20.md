## Applied Changes

### Change 1: Direct slot-index grapple carry lookup in blob worker

**File:** `server/src/net/websocket_server.c`  
**Function:** `build_shared_blobs_from_snapshot` (player entry loop)

**What:** Removed an inner `for (_si = 0; _si < WS_MAX_CLIENTS; _si++)` scan that searched for the matching `players[]` slot on every active player entry build. The outer loop index `p` already corresponds to the websocket player slot (`snap->players[p]` is copied from `players[p]`), so grapple carry weight now uses `players[p]` directly.

**Evidence:** Up to 100 strcmp-free pointer comparisons eliminated per active player per blob build. At 100 players × 30 Hz that is up to 300 000 comparisons/s removed from the blob worker hot path.

**Risk:** None. Guard `players[p].active && players[p].player_id == snap->players[p].player_id` preserves correctness if slot state ever diverges.

---

### Change 2: O(1) on-ship world position lookup for players and NPCs

**File:** `server/src/net/websocket_server.c`  
**Functions:** `SnapShipLut` helpers, `build_ships_blob_from_snapshot`, `build_shared_blobs_from_snapshot`

**What:** Extended `SnapShipLut` with a `ship_id → aoi_ship_index` map (`aoi_idx[]`, 128 KB) populated while building ship JSON. Player and NPC AOI world coordinates for entities aboard ships now use `snap_ship_lut_aoi_pos()` (O(1)) instead of scanning `aoi_ship_id[]` (O(ship_count) per entity).

**Evidence:** With 37 NPCs and 2 ships idle, this removes up to 74 linear scans per blob build; worst case at 512 NPCs × 200 ships is ~102 400 comparisons/tick replaced by array lookups. Stale-entry guard checks `aoi_ship_id[idx] == sid`.

**Risk:** None. Positions still come from the same `aoi_ship_px/py` arrays written during ship JSON build; only the lookup path changed.

---

### Change 3: Track player-entry length from `snprintf` return value

**File:** `server/src/net/websocket_server.c`  
**Function:** `build_shared_blobs_from_snapshot` (player entry loop)

**What:** Capture `_pelen` from the main player JSON `snprintf` return value; append grapple fields via `memcpy` using tracked offsets. Removed three `strlen()` calls and a `strncat()` per active player with an active grapple.

**Evidence:** Same pattern as 2026-06-15/06-17 blob-worker `strlen` eliminations; avoids redundant scans of ~1–3 KB strings on the worker thread.

**Risk:** None. Output bytes unchanged; length clamping preserved.

---

### Change 4: Bounded `dynamic_companies` snapshot copy

**File:** `server/src/net/websocket_server.c`  
**Functions:** `blob_worker_submit_snapshot`, synchronous fallback snapshot assembly

**What:** Copy only `dynamic_company_count` contiguous entries instead of the full `MAX_DYNAMIC_COMPANIES` array when submitting blob work.

**Evidence:** Companies are append-only in slots `0..count-1`. Typical world has 0–few companies; avoids copying unused tail slots every tick on the submit path.

**Risk:** None. `dynamic_company_count` is already the authoritative bound used when serializing `co_json`.

## Proposed (Needs Human Review)

### A. `per_gs_pool` -Warray-bounds: signed offset wrap

`per_gs_pool[100][524288]` is written via `_goff` (int). When `_goff` exceeds `PER_GS_BUF`, `memcpy`/`snprintf` size arguments wrap to a huge positive `size_t`. A correct fix requires `size_t _goff` with a hard cap before every write, or dropping the client when the buffer is full. Deferred: correctness-critical send loop.

### B. Player entry dirty-flag cache

NPC entries already use `npc_key[]` to skip `snprintf` when fields are unchanged. Player entries (~3 KB each, inventory + grapple) are rebuilt every tick even for stationary docked players. A `player_key[]` cache would help AFK/docked players but must include grapple state and derived `speed_mult`/`can_sprint`. Deferred: worker-thread/global coupling needs careful audit.

### C. `placed_structure_count` linear scans in movement hot path

Multiple `for (_di = 0; _di < placed_structure_count; _di++)` scans per moving player for dock/shipyard lookup. A structure-ID hash would reduce this to O(1) but touches placed-structure data layout. Deferred.

### D. `view_radius` hardcoded to 5000

Client-reported view radius is parsed but the send path uses `const float _view_r = 5000.0f`. Changing this affects AOI fairness — gameplay decision, not an automatic perf fix.

### E. Tombstone / dropped-item full-array `memcpy` on blob submit

`Tombstone` and `DroppedItem` arrays use sparse slot allocation (first-free-slot), so bounded copy requires an active-count index or compaction pass. Arrays are smaller than ships/NPCs but still worth a future audit.

### F. `handle_island_collisions` / `handle_ship_collisions`

Correctness-critical physics paths in `simulation.c`. No safe automatic change without gameplay review.
