## Applied Changes

### Change 1 — `build_ships_blob_from_snapshot`: local SimpleShip ID lookup eliminates O(n²) scan

**File**: `server/src/net/websocket_server.c`

**What**: For each of up to `sim_ship_count` (≤ 200) sim-ships, the blob worker previously performed a linear scan of `snap->ships[]` (≤ 200 SimpleShip entries) to find the matching SimpleShip by ID. This was an O(sim_ship_count × simple_ship_count) loop executed every blob build.

A local `const SimpleShip* _ss_lut[SHIP_ID_LOOKUP_SIZE]` lookup table (512 pointers, 4 KB stack) is now built once with a single O(simple_ship_count) pass before the outer sim-ship loop. Each sim-ship lookup is then O(1).

**Evidence / math**:
- Worst-case old: 200 × 200 = 40 000 comparisons per blob build
- At 30 Hz: 1 200 000 comparisons/sec in the blob worker thread
- New: 200 (table build) + 200 (lookups) = 400 operations per build → 12 000 ops/sec
- Savings: ~99.7% reduction in the inner loop, or ~38 000 saved comparisons per blob build

Ship IDs are `uint16_t` (`entity_id`); on a single server `ship_id = ship_seq` (1–255, uint8_t range). The existing `g_ship_by_id[SHIP_ID_LOOKUP_SIZE]` pattern (SHIP_ID_LOOKUP_SIZE = 512) was already used in `find_ship()` for the same purpose; this change brings the blob worker path to the same O(1) quality.

**Risk**: Zero — purely mechanical replacement of O(n) scan with table lookup; no semantic change.

### Change 2 — Tombstone JSON build loop: replace per-tombstone ship scan with O(1) lookup

**File**: `server/src/net/websocket_server.c`

**What**: In `build_shared_blobs_from_snapshot`, the tombstone serialization loop previously performed a linear scan of `snap->ships[]` (up to `snap->ship_count` = 200 entries) for each active tombstone whose `ship_id != 0`, in order to find the parent ship's position and rotation for the local→world coordinate transform.

A separate `_tmb_ss_lut[SHIP_ID_LOOKUP_SIZE]` lookup table is now built once at the start of `build_shared_blobs_from_snapshot` with an O(simple_ship_count) pass, and the inner scan is replaced with a single O(1) lookup.

**Evidence / math**:
- Worst-case old: MAX_TOMBSTONES (64) × MAX_SIMPLE_SHIPS (200) = 12 800 comparisons per blob build
- At 30 Hz: 384 000 comparisons/sec
- New: 200 (table build) + 64 lookups = 264 operations per build → 7 920 ops/sec
- The table-build cost (200 iterations) is shared with the tombstone loop and amortised over all tombstones in that pass.

**Risk**: Zero — same O(1) pattern as the existing AOI cache lookups for players and NPCs; no behavioral change.

## Proposed (needs human review)

The following issues were identified but NOT changed because they require judgment about correctness, involve gameplay-affecting constants, or need a more invasive refactor:

1. **`per_gs_pool` -Warray-bounds / `_goff` overflow risk** (`websocket_server.c` ~L12266):  
   `_goff` is a `signed int` accumulator into `per_gs_pool[WS_MAX_CLIENTS][PER_GS_BUF]`.  
   When `_goff` grows beyond `PER_GS_BUF` (524 288 bytes), subsequent `snprintf` calls receive a wrapped `size_t` argument (from `PER_GS_BUF - _goff` with a signed negative) and the compiler diagnoses it as `-Warray-bounds`. The code appears to guard against this but the unsigned promotion makes the guard insufficient.  
   **Fix path**: Convert `_goff` to `size_t`, add an explicit clamp `if (_goff >= PER_GS_BUF - 1) _goff = PER_GS_BUF - 2;` before each `snprintf`, and verify that `PER_GS_BUF` (512 KB) is actually large enough for worst-case entity counts. Needs human review for correctness.

2. **Snapshot tier timestamp reset bug (already fixed in previous run, confirmed)**:  
   The `tier_sent[]` guard (snapshot.c lines 148–297) is in place and correct. No further action needed.

3. **`g_npcs_dirty` flag unused in hot path** (`websocket_server.c` L165):  
   The flag is set when NPC state changes but the blob worker rebuilds `npc_entry[]` every tick unconditionally. Since NPC world positions depend on parent ship positions (updated each tick), the entry content changes every tick for ship-mounted NPCs. Caching requires per-NPC dirty tracking; left for human review.

4. **`view_radius` parsed from client but hardcoded 5000 used in send path** (`websocket_server.c` ~L12302):  
   `_view_r2 = 5000.0f * 5000.0f` is used for AOI filtering; the client-supplied `view_radius` is read but not used for this computation. Enabling the client value could allow variably-sized AOI windows, but raises fairness concerns. Deferred.

5. **Tombstone ship-lookup in `tombstone_world_pos()` (live path)** (`websocket_server.c` ~L1958):  
   The live path (called for range checks in player interactions) uses `find_ship()` which is O(1) via `g_ship_by_id[]`. Already fast; no action needed.

6. **`snapshot.c` delta baseline entity scan** (snapshot.c L239–244):  
   The UDP snapshot path performs an O(baseline_count) linear scan per subscribed entity to find the matching baseline. With `MAX_ENTITIES_PER_SNAPSHOT` entities this is O(n²). The UDP path does not appear to be active at runtime in the current deployment (no UDP snapshot send stats seen in logs). Deferred.

7. **`handle_island_collisions` / `handle_ship_collisions` (simulation.c)**:  
   Island collision is O(islands × ships × hull_vertices) and ship-ship CCD is O(ships²). Both use broad-phase early-exit guards and are correctness-critical; no safe automatic change.

## Summary

Two mechanical optimisations applied in the blob worker serialisation path:

| Change | File | Before | After |
|--------|------|--------|-------|
| SimpleShip ID lookup | websocket_server.c `build_ships_blob_from_snapshot` | O(sim_ships × simple_ships) scan | O(simple_ships) table build + O(1) per sim_ship |
| Tombstone ship lookup | websocket_server.c `build_shared_blobs_from_snapshot` | O(tombstones × simple_ships) scan | O(simple_ships) table build + O(1) per tombstone |

At idle (2 ships), the difference in blob-worker build time is below measurement noise (47–84 µs). The savings scale linearly with ship count and are most visible at ≥50 ships.
