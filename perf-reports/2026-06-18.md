## Applied Changes

### Change 1: Replace inline ship scan with `find_ship()` in per-player tick loop

**File:** `server/src/net/websocket_server.c`  
**Location:** Inside `for (uint16_t i = 0; i < global_sim->player_count; i++)` (was lines 14195–14206)

**What:** Every tick, each active player's ship was located by a `for s in 0..ship_count` linear scan comparing `ships[s].ship_id`. The `find_ship()` helper already maintains a `g_ship_by_id[512]` pointer cache (populated lazily) that makes this O(1) for all current ship IDs (1–255, well within the 512-slot table). Ship IDs never collide or wrap in a single session, so the cached pointer is always valid for the lifetime of the ship.

**Evidence:** With `ship_count = 5` the inline scan takes up to 5 comparisons per player per tick. `find_ship()` resolves in 1 array dereference + 2 equality checks (active + id). At 100 players × 30 Hz the scan budget is 15 000 comparisons/s vs. ≤ 600 lookups/s — a 25× reduction in worst-case comparison count for this path.

**Risk:** None. `find_ship()` returns a pointer into the same `ships[]` array; the pointer remains valid throughout the tick. No semantic change to any downstream ship-walking logic.

### Change 2: Eliminate redundant `player_inventory_weight()` in dock-walking branch

**File:** `server/src/net/websocket_server.c`  
**Location:** Inside `else if (ws_player->on_dock_id != 0)` branch of the per-player movement block

**What:** A prior commit (2026-06-14) hoisted `player_inventory_weight(ws_player)` to `_ws_inv_kg` at the top of each player's tick block, computing `_ws_carry_cap` and `_ws_carry_r` once for all branches. However, the dock-walking branch (`on_dock_id != 0`) was not updated — it still called `player_inventory_weight()` again and recomputed identical `_carry_cap` / `_carry_ratio` values. This fix replaces those three lines with references to `_ws_carry_r` and `_ws_sprint_ok` already in scope.

**Evidence:** `player_inventory_weight()` iterates all 16 inventory slots with mass lookups. `ws_player->stat_weight` and inventory contents do not change between the top-of-block hoist and the dock-walking branch within the same tick. The expressions are algebraically identical (same formula, same inputs). Removing the duplicate saves 1 full inventory scan + 2 FP divides per docked player per tick (up to 30 Hz × 100 players = 3 000 saved scans/s at capacity).

**Risk:** None. Pure substitution of equivalent values within the same lexical scope and tick.

## Proposed (Needs Human Review)

### A. `per_gs_pool` -Warray-bounds: signed offset wrap

`per_gs_pool[100][524288]` is written via `_goff` (int). When `_goff` exceeds `PER_GS_BUF = 524288`, the `snprintf` size argument becomes `(size_t)(524288 - _goff)` which wraps to a huge positive integer, allowing out-of-bounds writes that corrupt adjacent struct fields and produce malformed WebSocket frames. The compiler correctly flags this as undefined behaviour. A correct fix requires either: (a) switching `_goff` to `size_t` and capping before each write, or (b) adding a `if (_goff >= PER_GS_BUF) { drop client; continue; }` guard. Both require careful audit of all `_goff` increment sites. Deferred: correctness-critical and touches the per-client send loop.

### B. `ship_count`-bounded snapshot copies

`blob_worker_submit_snapshot()` copies `sizeof(ships)` = 200 × ~9 KB ≈ 1.8 MB per tick into the worker job, regardless of `ship_count`. With 5 active ships, 195 empty slots are copied needlessly. This could be changed to `ship_count * sizeof(SimpleShip)`, but requires confirming that: (1) all active ships are always contiguous in `ships[0..ship_count-1]` with no active-flag holes, and (2) the blob worker and fallback path always bound their ship iteration by `snap->ship_count`. Deferred for human review of the ship allocation/removal paths.

### C. World-NPC array copy in blob snapshot

`memcpy(job.world_npcs, world_npcs, sizeof(world_npcs))` copies 512 × sizeof(WorldNpc) every tick regardless of `world_npc_count`. If WorldNpc is large, this is significant overhead. Same concern as (B): requires verifying contiguity of the active-NPC prefix. Deferred.

### D. `placed_structure_count` linear scans in dock-walking hot path

Inside the per-player movement block, multiple `for (_di = 0; _di < placed_structure_count; _di++)` scans search for a dock structure by `id`. With many placed structures these accumulate O(player_count × placed_structure_count) iterations per tick. A structure-ID → index hash would make each lookup O(1), but modifying the placed-structure data structures is a larger refactor. Deferred.

### E. `view_radius` hardcoded to 5000 despite being parsed from client

The per-client AOI filter uses `view_radius = 5000` unconditionally (client value parsed but discarded). Changing this value affects which entities each client receives — fairness/gameplay decision, not a pure perf fix. Reported only.
