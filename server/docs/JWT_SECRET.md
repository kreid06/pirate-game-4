# JWT_SECRET setup

The auth server signs login tokens with **HMAC-SHA256**. The game server verifies those tokens using the same shared secret. Both processes read `JWT_SECRET` from the environment.

If `JWT_SECRET` is missing, too short (< 32 characters), or still a placeholder value, the game server **refuses to start**.

---

## Generate a secret

Use a cryptographically random value â€” at least 32 characters (256 bits recommended):

```bash
openssl rand -hex 32
```

Example output (do **not** reuse this; generate your own):

```
a1b2c3d4e5f6789012345678901234567890abcdef1234567890abcdef123456
```

## Local development

### Option A â€” export in your shell (quick test)

```bash
export JWT_SECRET="$(openssl rand -hex 32)"

# Game server (C)
cd server && make && ./bin/pirate-server

# Auth server (Node) â€” separate terminal, same secret
cd server/auth && npm install && npm run build
JWT_SECRET="$JWT_SECRET" npm start
```

### Option B â€” `auth.env` file (recommended)

Create `server/config/auth.env` (this path is gitignored):

```bash
mkdir -p server/config
openssl rand -hex 32 | awk '{print "JWT_SECRET=" $0}' > server/config/auth.env
echo "AUTH_PORT=3001" >> server/config/auth.env
echo "CORS_ORIGINS=http://localhost:5173" >> server/config/auth.env
chmod 600 server/config/auth.env
```

Load it before starting either server:

```bash
set -a
source server/config/auth.env
set +a

cd server && ./bin/pirate-server
```

In another terminal (same `source` step):

```bash
cd server/auth && npm start
```

## Production (VPS)

`server/setup-server.sh` generates a random secret and writes `/opt/pirate-game/config/auth.env`:

```env
AUTH_PORT=3001
JWT_SECRET=<auto-generated 64-char hex>
CORS_ORIGINS=https://your-game-domain.com
```

Both systemd units load that file:

- `pirate-server.service` â†’ game server (C)
- `pirate-auth.service` â†’ auth server (Node)

After editing the file:

```bash
sudo systemctl daemon-reload
sudo systemctl restart pirate-auth pirate-server
```

Verify it is set (shows the variable name only if you grep carefully â€” avoid pasting secrets into chat):

```bash
grep -q '^JWT_SECRET=.' /opt/pirate-game/config/auth.env && echo "JWT_SECRET is set"
```

## GitHub Actions deploy

CI writes `auth.env` on the server from the repository secret **`AUTH_JWT_SECRET`**.

1. On the VPS, read the generated value:
   ```bash
   grep JWT_SECRET /opt/pirate-game/config/auth.env
   ```
2. In GitHub: **Settings â†’ Secrets and variables â†’ Actions**
3. Add or update **`AUTH_JWT_SECRET`** with the same value (no quotes, no `JWT_SECRET=` prefix).

The deploy workflow copies this into `/opt/pirate-game/config/auth.env` on each deploy.

## Rules

| Requirement | Why |
|-------------|-----|
| **Same secret** on auth + game server | Game server verifies signatures the auth server creates |
| **â‰¥ 32 characters** | Enforced at startup in `server/src/main.c` |
| **Never commit** the real secret | `server/config/auth.env` is in `.gitignore` |
| **Rotate if leaked** | Anyone with the secret can forge valid login tokens |

## Troubleshooting

| Symptom | Fix |
|---------|-----|
| `FATAL: JWT_SECRET environment variable is not set` | Export `JWT_SECRET` or add `EnvironmentFile=` / `source auth.env` |
| `JWT_SECRET is still the default placeholder` | Run `openssl rand -hex 32` and replace the value |
| `JWT_SECRET is too short` | Use at least 32 characters (`openssl rand -hex 32` gives 64) |
| Auth works but game rejects login | Auth and game servers have **different** secrets â€” sync them |
| `JWT_SECRET not set â€” refusing token` (runtime log) | Game server started without `EnvironmentFile`; restart with env loaded |
